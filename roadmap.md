# roadmap.md — 인터랙티브 플레이 앱 개발 계획

> 이 문서는 토큰/비용 계측(v12 '클리포트 억지력' + 세피로트 분석)으로 **실측 데이터에 근거해** 도출한 다음 개발 우선순위다.
> 규약은 [CLAUDE.md](CLAUDE.md)·[worldstate.md](worldstate.md)·[tonemanner.md](tonemanner.md) 참조. 대상 산출물은 `lorebook-play.html`(빌드: `build_play.py`/`play_template.py`).

## 0. 현황 (v6 → v12)

- v6 인터랙티브 플레이 앱(클라이언트 단독·BYO 키, 다중 프로바이더, WorldState 엔진, `state` 델타).
- v7~v9 시작 UI(팩션 탭·슬롯·핸드)·스테이터스 마스터-디테일·플럼 테마·모델 카드 선택·입력바 칩/빠른액션.
- v10 캐릭터 캐논 프로필(`CANON.raw`)을 LLM 프롬프트에 복원(`charDossier`)+장면 등장 인물 자동 감지.
- v11 위키를 별도 메뉴로 병합(격리 iframe, 이미지 부모 공유, 플럼 통일).
- v12 토큰·원화 비용 HUD + 세피로트 입력 분석(`assembleContext` 세그먼트 태깅, provider usage 캡처, 실시간 환율).

## 1. 계측으로 드러난 사실 (근거)

v12 세피로트 분해의 대표 측정(주인공+동행 1, 2턴, mock 기준 추정):

| 세피로트 | 출처 | 비중 | 함의 |
|---|---|---|---|
| 케테르&호크마 | 등장인물 캐논(동행/장면 프로필 raw) | **~41%** | 인물 프로필이 최대 토큰 소비 |
| 티페리트 | 페르소나(주인공 프로필 raw) | **~34%** | 주인공 프로필도 거대 |
| 비나 | 시스템 프롬프트(문체 규칙) | ~17% | 고정 오버헤드 |
| 헤세드/게부라 | 최근 대화(유저/AI) | 합 ~2% | 초반엔 작으나 **턴이 쌓이면 선형 증가** |
| 네짜 | 로어/세계 상태 | **~0.2%** | **세계관·로어가 사실상 미주입** |
| 호드/예소드/말쿠트 | 호감도·장착·프로토콜 | 각 1~2% | 소량 |

**핵심 결론 3가지:**
1. **비용의 ~75%가 인물 프로필 raw**(케테르+티페리트). 매 턴 전체 원문을 반복 전송 → 캐싱/요약 없이는 회차가 길어질수록 누적 비용이 큼.
2. **네짜(로어)≈0** — 위키엔 12개 로어 문서(마법소녀 체계·괴이·마법소녀청·시고쿠의 비극 등)가 있는데 **플레이 LLM은 이를 전혀 못 본다.** 서사 깊이의 최대 공백이자, 채울 가치가 큰 빈 세피라.
3. **대화 히스토리(헤세드/게부라)가 무한 증가** — `convo` 전체를 매 턴 전송. 장기 플레이 시 비용·지연·맥락 한도 문제. 호드(장기 기억)가 실질적으로 비어 있음(현재 호감도 밴드뿐).

## 2. 우선순위 로드맵

### P0 — 비용/품질 즉효 (다음 사이클)

#### P0-1. 프롬프트 캐싱 (Anthropic prompt caching) — 비용 절감 최대치
- **문제**: 인물 프로필+시스템 프롬프트(=비용 ~92%)가 턴마다 동일한데 매번 풀가로 과금.
- **접근**: `callLLM`(anthropic)에서 안정 블록(시스템 문체 규칙 + 인물 프로필)을 `cache_control:{type:'ephemeral'}`로 표시. 변동부(호감도/플래그/최근 대화)는 캐시 밖. 헤더 `anthropic-beta: prompt-caching` 필요 여부 확인(현행 모델 기본 지원 여부 점검 → claude-api 스킬/문서 확인).
- **영향**: 캐시 적중 시 입력 토큰 비용 최대 ~90% 절감(반복 프로필). HUD에 `캐시 적중` 표시(usage의 `cache_read_input_tokens`).
- **작업**: `callLLM` 본문 구조화(system을 배열 블록으로), usage 캡처에 캐시 토큰 추가, 비용식에 캐시 단가 반영. 중간.
- **검증**: 동일 회차 2턴 연속 시 2턴째 `cache_read`>0, 비용 급감 확인.

#### P0-2. 로어 컨텍스트 주입 (네짜 채우기) — 서사 깊이
- **문제**: `CANON.lore`(12문서)가 플레이 프롬프트에 없음 → 세계관 일관성·깊이 부족.
- **접근(RAG-lite, 오프라인)**: 키워드/엔티티 매칭으로 **현재 장면 관련 로어만** 선별 주입. ① 로어 문서별 키워드 인덱스(이미 `lore[].keys` 존재) ② 최근 대화+등장 인물에서 토큰 추출 → 매칭 상위 1~2개 문서의 **요약**을 `assembleContext`에 `netzach` 세그먼트로 주입(전문 아님, 토큰 절제). 항상 주입할 핵심 1개(예: 세계 규칙 요약)는 캐시 블록에.
- **영향**: 네짜 0%→유의미, 괴이·마법소녀청·시고쿠 등 설정 반영. 토큰은 요약·선별로 제어.
- **작업**: 로어 요약 사전(빌드시 생성 또는 수기) + 매칭 함수 + `assembleContext` 주입. 중간.
- **검증**: 시고쿠/괴이 언급 시 관련 로어가 `netzach`에 등장, 세피로트 분석에 반영.

#### P0-3. 대화 요약/장기 기억 (호드 채우기) — 무한 증가 차단
- **문제**: `convo` 전체 전송 → 장기 플레이 비용·한도 폭증.
- **접근**: 슬라이딩 윈도 + 요약. 최근 N턴은 원문, 그 이전은 **누적 요약 1블록**(`W.summary`)으로 압축. 요약은 (a) 주기적 보조 LLM 호출 또는 (b) 우리가 이미 만드는 `W.recap`(첫 문장) 확장. 요약/recap을 `hod` 세그먼트로.
- **영향**: 헤세드/게부라 상한 고정, 호드에 실질 내용. 장기 회차 가능.
- **작업**: 윈도/요약 로직 + 세이브 포함 + 토큰 회계 반영. 중간.
- **검증**: 20턴 플레이 시 입력 토큰이 선형 폭증하지 않고 평탄화.

#### P0-4. 예산·경고 + 출력 제어
- **문제**: 비용 상한·경고 없음. `max_tokens` 고정(2048).
- **접근**: 설정에 회차/일일 한도(클리포트 또는 ₩), 초과 시 HUD 경고·전송 확인. `max_tokens`·응답 길이 프리셋(짧게/보통/길게) UI. 모델별 예상 단가 칩에 표시.
- **작업**: 설정 UI + `turn` 가드 + HUD 경고. 소~중.
- **검증**: 한도 근접 시 경고, 초과 시 확인 프롬프트.

### P1 — 정확도·경제 정교화

#### P1-1. 토큰 추정 정확도
- 현재 `estTokens` 휴리스틱(한/CJK 0.6, 기타 0.27) → 구간 비율만 추정. 개선: provider `count_tokens`(Anthropic) 사용 또는 빌드시 경량 토크나이저 테이블 인라인. 세피로트 구간 정확도 향상.

#### P1-2. 프로필 토큰 다이어트 (케테르/티페리트 축소)
- charDossier가 `raw` 통째 → 옵션화: ① 핵심 섹션만(Speech/Combat/Personality/Core) ② 빌드시 인물별 **요약본** 생성해 기본 사용, 상세는 필요 시. ③ 동적 상세도(주인공=풀, 장면 단역=요약). 캐싱(P0-1)과 병행 시 효과 큼.
- **영향**: 비용 75% 구간 직접 절감. 단, v10에서 "방대한 설정 복원"이 목적이었으므로 **품질 저하 없이** 요약 품질 검증 필수.

#### P1-3. 비용 대시보드 확장
- 세피로트 모달에 **출력측**(말쿠트 너머) 비용, **회차 누적 분해**, 캐시 절감액, 모델 전환 시뮬레이터("이 회차를 Haiku로 했다면 ₩X"). 단가 자동 갱신(가격 API/주기 점검).

### P2 — 월드/게임성 심화 (worldstate.md 실현)

- **세계 플래그·영역 동적화**: territory stability 변화·백귀야행 트리거가 실제 서사/네짜에 반영(현재 거의 정적).
- **세계의 기억(NG+) 루프 완성**: 회차 종료 발급 → 다음 회차 해금/장착이 분기에 실제 영향(현재 골격만).
- **관계 다축(trust/respect/romance) 심화**: 임계 초과 시 행동 프로파일 스왑(나츠메 '운명의 상대' 등 캐논 분기) 서사 주입.
- **선택지/액션 → 사전정의 델타 매핑**(worldstate §4-2b), 분기 가시화.

### P3 — UX·운영

- 세이브 관리(이름·삭제·export/import 일원화), 회차 요약 카드.
- 모바일 레이아웃(레일/HUD/모달 반응형), 접근성.
- 위키↔플레이 연동(위키 인물 카드에서 "이 인물로 시작" 등).
- 온보딩(키 입력 가이드, mock 튜토리얼 1회차).

## 3. 비용·토큰 경제 (참고)

- 단가표(`PRICES`, USD/1M, 근사치): opus-4-8 5/25 · sonnet-4-6 3/15 · haiku-4-5 1/5 · gemini-2.5-pro 1.25/10 · gemini-2.5-flash 0.30/2.5 · gpt-4o 2.5/10 · gpt-4o-mini 0.15/0.6.
- 환율: 실시간 fetch(jsdelivr 통화 API→open.er-api 폴백→캐시→기본 1400), 24h 캐시.
- 1 토큰 = 1 클리포트 억지력. 비용 = `(in/1e6·pin + out/1e6·pout)·FX`.
- **레버리지 순위**: 캐싱(P0-1) ≳ 프로필 다이어트(P1-2) > 히스토리 요약(P0-3) > 모델 선택(저가 모델 권장 UI).

## 4. 리스크·제약

- **단일 HTML·BYO 키·vanilla** 유지(외부 런타임 의존은 LLM 엔드포인트 + 환율 fetch뿐, 둘 다 폴백). 캐싱/요약은 프로바이더별 가용성 차 → anthropic 우선, 타 프로바이더는 폴백.
- 프로필 다이어트·요약은 **품질 회귀 위험** → 별도 검증 패스(평어체·말투·관계 보존) 필수.
- 토큰 추정은 표기상 '추정'임을 유지(총량은 실 usage, 구간은 추정).

## 5. 다음 액션 제안

1. **P0-1 프롬프트 캐싱** + **P0-2 로어 주입**을 한 사이클로(비용↓·품질↑ 동시, 세피로트에 즉시 반영).
2. 이어서 **P0-3 히스토리 요약** + **P0-4 예산/경고**.
각 항목은 `build_play.py --report`/Playwright(mock) + 세피로트 분석 회귀로 검증.
